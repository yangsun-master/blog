Dependency Injection

Last updated: 2020-02-02

Dependency injection is supper useful. The constructor dependency injection is like a magic. What’s secret behind it? Let’s take a deep look.

Background

Why dependency injection is useful? The most important benefit is decouple. Class A reference class B, because A relies on B’s functionality, simply saying A calls B’s function. However, B doesn’t come out of blue. A has to know how to create or obtain B. But actually, A doesn’t really care about it. Even more, if B’s construction logic changed, it’s a breaking change to A. Dependency injection is a way to decouple.

How does dependency injection work? Here’s a full example: Dependency injection into controllers in ASP.NET Core

A short summary is:

  1. Define the class B and interface IB
  2. Declare IB as A’s constructor parameter, A save it as a member variable.
  3. Register IB with B in the startup

In the below example, A->B->C, the dependency chain is well constructed by the dependency injection library. Good job! It’s like a magic. I want to know the secret.

Dependency Injection deep dive

Overview

The question is: given a type at runtime, how to create the instance? The source code of the dependency injection library is on Github: Dependency injection library. After reading the code, I found the library uses three ways to dynamically create instance for runtime type. There’re three ways not one, it’s because the .Net runtime version. The code is here: ServiceProvider:

  1. Expression Tree
  2. Dynamic Method
  3. Reflection

The underlying mechanism of the 1st and the 2nd approach are same, they’re equivalent to new operator(var a = new A();), so they’re the most efficient. The 3rd one is reflection, which is less efficient.

Intermediate Language (IL)

Intermediate Language (IL) is a concept in .NET. C# is high level programing language. C# compiler compiles C# code into IL code, and stored in the assembly, which may be a dll or exe file. ILSpy is a tool to view IL code inside the assembly. .NET CLR (Common Language Runtime) loads the assembly, JIT(Just-In-Time) compiles the IL code into native code. The native code are native machine instructions, and executed by machine. Here’s the full explanation Introduction to the C# language and the .NET Framework.

Back to this topic, the idea is to generate IL code to new an instance for a type at runtime, which is equivalent to write C# code to new an instance for a known type. .Net framework provides dynamic method and expression tree to generate IL code at runtime.

C# -> IL -> native code

new operator’s corresponding IL code is newobj.

Dynamic method

Here’s the steps to use Dynamic Method, and the full explanation of the document.

  1. Create a Dynamic Method object
  2. Generate IL codes as the body of the Dynamic Method.
  3. Create a delegate to invoke the Dynamic Method.

Dynamic Method is efficient. The disadvantage is the code maintenance. IL is stack-based language (wiki). The variables are defined on the stack, the instructions operates the variables on the stack. It’s easy for machine (CPU and memory) to understand, but it’s not easy for human to understand. Human’s brain doesn’t work like CPU and memory. :P. The second is generating wrong IL code, the compiler won’t complain. The code is here: ILEmitResolverBuilder

Expression Tree

The previous generate IL code approach is a way like we’re the compiler, we translate the programing language to the code machine can execute. All the work are done by us. The Expression Tree approach is a way like we build the syntax tree, and let the compiler generate the machine code. This approach is also efficient, and the code maintenance is also improved. Here’s the document. The code is here: ExpressionResolverBuilder

Reflection

Reflection is different from the previous two. The approach is to call the Invoke function on the ConstructorInfo. Digging into the call stack, the last function call is RuntimeMethodHandle::InvokeMethod, which is implemented by .NET runtime internal (MethodImplOptions.InternalCall). The code is simple, but the performance is not good as the previous two. By spying the implementation, I found the implantation isn’t simple, it constructs a Signature, perform security check on demand, and argument’s value type check. All these check contribute to slow down performance. The code is here: CallSiteRuntimeResolver.
